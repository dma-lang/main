01Product overview & users

An internal, single-tenant web application for ~10–30 expert users (Zennify pillar leads, consultants, data architects). It keeps the four-pillar capability catalogue provably current by ingesting two corpora — the four-pillar capability workbooks (across versions) and the canonical Jira story corpus — reconciling versions into one canonical model, provisioning a persistent relational database per version via an admin-controlled mapping step, and serving trust-first read & reasoning surfaces over that data.

Non-negotiable product principle — trust-first. Every AI output carries four pieces of metadata: a claim label (FACT · INFERENCE · HYPOTHESIS · CEILING_ESTIMATE), a source tier (T1–T5), an evidence reliability score (ERS), and a reasoning chain the user can open. These are not decoration — they gate commits (§9).

02System architecture

A single Cloud Run service hosts the FastAPI backend and the built React SPA behind one container. Asynchronous work is split into Cloud Run Jobs. The system of record is a relational database; object storage holds raw uploads, snapshots and signed exports.

React SPAVite · router v6 · Zustand · TanStack Query
Cloud RunFastAPI + static SPA, one container
Cloud SQLPostgreSQL 16 · pgvector · full-text
Jobs / GCSingest, carry-forward, digest · buckets

Why relational. The catalogue is inherently relational (subcaps↔platforms, stories↔subcaps, offerings↔subcaps). The mapping studio defines those relations as foreign keys; the same Postgres engine serves hybrid retrieval through pgvector (dense) and tsvector (lexical) alongside ordinary SQL filters — one store for structure, search and provenance.

5.2 Data plane (per version) — cat_v5.*, cat_v7.*

Each version owns a Postgres schema namespace holding the canonical entities for that version: subcap, category, capability, l4_feature, persona, use_case, maturity_descriptor, and the deduplicated entities (offering, data_product, l3_platform, theme, agent) with their link tables (subcap_platform, subcap_persona, offering_subcap, …). The mapping studio generates this schema; the app reads the active version by name.

Why schema-per-version (not a version column)

Physical isolation: re-provisioning or reverting a version never risks another version's rows, and one version's admin-defined relations can't constrain another. Cross-version questions (lineage, diff, carry-forward) are answered by the control plane, which holds the crosswalk (_R1_Source_Reference) and the carry-forward results.

The real v7 relations the prototype's mapping studio shows (and that provisioning realises) — these are the authoritative relation_def rows:

5.3 Handling both catalogue versions (v5 & v7)

The app must serve both the legacy v5 and the current v7 catalogue from the same code, switching by the header version toggle. They are physically separate data-plane schemas (cat_v5, cat_v7); the control plane reconciles them.

Engineering rules: (1) each subcap keeps one identity across versions via the _R1_Source_Reference crosswalk in the control plane; (2) every count, diff and lineage payload is version-qualified (R3); (3) the Diff viewer reads the crosswalk to compute added/removed/modified between any two versions; (4) carry-forward (§7) is what moves the v5-mapped Jira corpus onto v7; (5) reverting a version restores its schema snapshot but never rewinds the evidence-corpora timestamps (the record of "what was known when" is preserved).

06Schema-mapping engine (FR-2)

What makes the system resilient to however a future catalogue version structures its workbook. Runs on upload, admin-controlled end to end. A new catalogue structure must never require a code release.

StepWhat happensEngineering note
DetectIdentify source layout, pick parsing adapterDeterministic; unknown layout → manual mapping, not a failure
Auto-mapPropose a canonical field per source field + confidenceEmbedding + heuristic match on header names & sampled values
Confirm / overrideAdmin accepts high-confidence, resolves flaggedEvery field ends confirmed, custom, or ignored
Custom fieldsAdd columns with no canonical homeNothing in source silently discarded
RelationsDefine FK relations for this version from the UIWrites relation_def rows (see §5.2)
CascadePropagate compatible relations to other versionsOnly where fields exist and are type-compatible
Apply / provisionWrite schema, create tables + FKs, load rowsTransactional & idempotent; re-runnable to correct a mapping
# ingestion/provision.py — provisioning is ONE transactional, idempotent op keyed on version_id
def bring_version_online(version_id, workbooks):
    layout    = detect_layout(workbooks)                 # "v5" | "v7" | unknown→manual
    proposal  = automap_fields(layout, CANONICAL_MODEL)  # [(src, canon, confidence, status)]
    mapping   = await admin_confirm(version_id, proposal)
    mapping  += admin_custom_fields(version_id)
    relations = admin_relations(version_id) + cascade_compatible(...)
    with txn():
        create_schema(f"cat_{version_id}")
        for e in CANONICAL_MODEL: create_table(e, mapping[e] + custom_cols(e))
        for r in relations:  add_foreign_key(r)
        load_rows(workbooks, mapping)
        register_ingest(version_id, status="provisioned")   # durable for ALL users (D10)

07Story pipeline & carry-forward (FR-3, FR-4)

The canonical story source is the Jira Full Story Catalog (14,406 real stories, 87 mapped subcaps) — the analysis set. It supersedes the workbook story tabs and is persisted in the control plane. Synthetic / use-case-derived rows are loaded but excluded from analysis by default; they surface only when a trend flags a subcap as emergent, then labelled provisional.

Carry-forward correlation — deterministic first, semantic second

Because the catalog was mapped against v5, carry-forward maintains those mappings on the active version: (1) normalise the id, stripping any subvertical suffix (e.g. P3C1.8.CL2 → P3C1.8); (2) bridge with the target version's crosswalk; (3) confirm similarity with the catalogue-tuned embeddings.

Thresholds: ≥ 0.86 → carried automatically · 0.70–0.86 → one-click admin confirm · < 0.70 → unmapped with a nearest-neighbour proposal. Idempotent & re-runnable; results written to the control plane. Ships as carry_forward_subcap_mappings.py.

One embedding space backs story matching, SOW matching and carry-forward, so a subcap means the same thing whether matched, retrieved, or carried between versions.

08AI / ML subsystem (FR-7…FR-17)

Grounded and tiered. Answers and matches are retrieved from stored catalogue and evidence, never from model memory. Model strength is matched to leverage.

JobModel tierUse
Classification / extractionGemini FlashNews tagging, event typing, story field extraction — high volume, low cost
Matching / reasoning loopGemini Pro / Claude SonnetSOW→subcap & story→subcap matching, impact scoring, reasoning steps
Synthesis / adversarial reviewClaude OpusQuarterly digest, benchmark adversary verdict — bounded volume
EmbeddingsCatalogue-tunedRetrieval, matching, carry-forward — one shared space

Retrieval is hybrid in one engine: lexical (tsvector+GIN) + dense (pgvector HNSW) + structured SQL filters (pillar, subvertical, claim label, source tier). The catalogue ontology itself is part of the retrieval corpus so chat is catalogue-grounded (FR-15).

Cost control: every model call is tagged (operation, pillar, subvertical, subcap, batch) for exact cost-by-feature. Alert at 80% of the monthly envelope; throttle non-essential ops at 90%; auto-route to a fallback model with a degraded-mode banner rather than failing. Frontend note the cost meter and degraded banner exist in the prototype and are admin-only.

09Eight validation gates (G1–G8)

Deterministic gates that sit between any AI proposal and any commit. They are code, not prompts — reproducible and auditable. A proposal failing any gate routes to the Change Flags inbox with the failing gate named; it is never auto-applied. Each run persists a validation_gate_run linked to its reasoning chain.

GateNameCheck
G1Identity & schemaTarget subcap resolves in the active version; payload validates against the canonical model
G2Evidence sufficiencyClaim has the minimum evidence its label requires; bare assertions rejected
G3Source-tier floorEvery cited source meets the minimum tier; a commit needs stronger sourcing than a note
G4Claim-label consistencyAsserted label matches evidence strength; no FACT on a single weak source
G5Similarity / groundingMatches & carry-forward meet the cosine floor; chat answers overlap retrieved context
G6Contradiction checkA proposal conflicting with committed evidence is flagged contradicted_evidence
G7Citation verificationEvery citation resolves to a stored, retrievable chunk; fabricated citations fail
G8Budget & rateOperation within cost envelope & rate limits; degraded-mode routing applied

10Knowledge graph (admin-only, D18)

Layer A — deterministic

A projection of committed structure: foreign keys and link tables become edges. Always trustworthy.

BELONGS_TO ADDRESSED_BY DELIVERED_BY MAPS_TO_OFFERING USES_PLATFORM TAGGED_THEME HAS_PERSONA HAS_USECASE EVIDENCED_BY

Layer B — AI-proposed, gated

Edges the structure doesn't state outright. Must pass G1–G8 and human approval in Change Flags before commit. Limited to SEMANTICALLY_SIMILAR (cosine ≥ 0.85) and SHARES_FEATURE. Cross-pillar dependency edges are out of scope (D16). No AI edge is ever written live — the proposer emits a pending_edge.

Routing & shell — global components

  • Auth gate: Login → first-run onboarding → app shell. Onboarding always precedes Mission control on a fresh session.
  • App shell: brand cell, header (pillar segmented control, subvertical & lens dropdowns, version toggle, is_admin toggle, change-flag bell, admin-only cost meter, theme toggle), and a 9-group sidebar (A Explore · B Catalogue tools · C Project validation · D Public intelligence · E Synthesis · F Lifecycle & competition · G Versioning & QA · H Reasoning & RAG · I Sandbox).
  • Merged Capability workbench: explorer + deep dive are one master-detail surface — left tree navigator (pillar→category→cluster→subcap, searchable), right focused subcap with 5 tabs (Overview · Maturity · Use cases · Delivery · Connections) and a completeness ring. Progressive disclosure keeps it un-overwhelming.
  • Cross-cutting trust surfaces: a reasoning-chain modal (opens from every "Show reasoning"), a subcap "quick-look" peek drawer, a consultant-reasoning-loop modal (capture thesis → AI adversarial challenge → gates → stage/commit), and a versioned commit modal.

Design system (binding)

Zennify DMA system: DM Sans only; calm teal/mint/lavender palette (#185F60 dark teal, #27BBAF brand teal, #1C4A4D ink, #F2F4F9 lavender, #3D81F6 blue, #FE9732 orange); sentence case everywhere except ALL-CAPS eyebrows & maturity labels; no gradients on content, no shadows on slides/cards beyond a hairline, no emoji; flat geometry; fades only (120–200ms). Light + dark themes both ship.

14State, caching & real-time

  • Server cache: TanStack Query keyed by [version, resource, filters]. The active version is part of every catalogue key so a version switch refetches cleanly.
  • Client state: Zustand for ephemeral UI (open tab, drawer, selected node). Filter state is URL-serialized (pillar, sv, lens, page, search) for deep links — mirror the prototype's hash params.
  • Heavy lists (stories 14k, use cases) are server-paginated & server-filtered — never ship the full set to the client (SLO §15).
  • Long jobs (provision, carry-forward, monthly cycle, digest) are async Cloud Run Jobs — the UI polls ingest_run.status / job status and shows progress, exactly like the onboarding stepper.
  • Chat is streamed; digest/what-if may be long — show the reasoning/progress affordances the prototype already has.

16Environments & hermetic dev

Three environments — dev (hermetic LLM mode: the full stack runs with no provider credentials, model stack stubbed deterministically), staging (real providers, synthetic admin), production. The same container image is promoted; only config/secrets differ. Provider availability is a config switch, not an architectural dependency. Tip the delivered prototype is effectively the hermetic-dev frontend — deterministic data, no provider calls — so it doubles as the dev-mode UI.

18Required document refinements

Gaps / inconsistencies found while reconciling the five source documents against the build. None block Phase 1; each needs a decision recorded.

#FindingRecommended resolution
R1Platform-name parsing in the source workbooks. The L3 platform cell is newline-delimited [L3-ID] Name with trailing (count: N); a naive parser captures multi-line garbage (codes + counts) into names. The prototype's first dataset hit this.Standardise the L3 cell parser: split on newline, strip the [L3-…] token and (count: N) suffix. Document the cell grammar in the Backend Schema doc so every ingester treats it identically.
R2Relation vocabulary drift. The TRD/KG prose uses matures_through / mapped_to; schema.sql's relation_type enum uses addresses (maturity) and custom (value chain). The prototype follows the SQL enum.Make schema.sql the single source for relation verbs; update TRD §9 prose to match the enum, or add the missing verbs to the enum explicitly.
R3Subcap count reconciliation. PRD/headline says ~851 subcaps; v5 vs v7 per-pillar counts differ (v7 adds the FC subvertical: +11). Diff/lineage must state which version a count refers to.Always qualify counts by version in copy and API. The control-plane crosswalk already supports this — surface it in the diff payload.
R4Carry-forward thresholds are uncalibrated (TRD §15). 0.86 / 0.70 are placeholders.Calibrate on the real v5→v7 run; keep them in config, not code so they tune without a deploy.
R5News "expected catalogue impact" classification is implied by the PRD (descriptor revision / new use case / net-new subcap) but not enumerated.Add an explicit catalogue_impact enum on the news/evidence model (the prototype demonstrates: Descriptor revision · New use-case candidate · Net-new subcap · Retire candidate · Watchlist/no-change) and let it drive the suggestion kind.
R6Source attributes on evidence. The PRD shows source tier but the surfaceable attributes (canonical name, type, URL, reliability/ERS, last-checked) aren't fully specified for the News/Sources UI.Define the evidence.source sub-object: {name,type,tier,url,ers,fetched_at}; the prototype's News watch shows the intended rendering.
R7Cost visibility scope. PRD treats cost as an admin/governance concern but the UI brief shows a persistent cost meter.Confirmed in the prototype: cost meter & per-thread cost are admin-only. Reflect this in the UI/UX brief.
R8Explorer vs deep-dive. The original UI/UX brief specified them as separate pages; usability testing on the prototype merged them into one workbench.Adopt the merged "Capability workbench" in the UI/UX brief as canonical; retain a deep-link (/subcap/{id}) that opens the workbench focused.
R9SOW corpus is unenriched.
