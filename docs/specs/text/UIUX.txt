## From brand to product

 The Zennify brand is calm, clean, modern and human-first. In a workbench where consultants commit catalogue changes on AI evidence, those four principles become concrete, enforceable rules.
 | Brand principle | Product implication | Anti-pattern prevented

 | Calm | Dense information discloses progressively: an irreducible summary is visible, depth on demand. No gradients, animation, or decorative chrome. | Overwhelming dashboards
 | Clean | White space is first-class. 8px base grid. Hierarchy from background-colour contrast, never drop shadows. | Cards with shadows fighting for attention
 | Modern | Consistent 6px radius on surfaces, 4px on tags. No pill shapes, no border-left accent bars. Colour carries meaning, not decoration. | Generic AI-looking UI
 | Human-first | Trust is the outcome. Every AI output shows claim label, source tier, and a reasoning-chain link. Irreversible actions get preview modals. | Black-box AI with no provenance

 1

#### Trust is the primary outcome

 Claim + tier chips render on every AI card, never hidden in overflow. The reasoning-chain link is always a visible affordance.

 2

#### Recognition beats recall

 Pillar palette, lifecycle colour, claim-label style and tier-chip shape are identical across all 27 pages. Orange always means the same thing.

 3

#### Progressive disclosure

 Card summary is one line plus chips; depth expands on demand. Modals for depth, not page hops.

 4

#### Reversibility

 Destructive actions are filled orange, never teal. Confirm modals list exact impact counts. Cancel has equal visual weight to confirm.

 5

#### The catalogue is the gravity well

 Any Sub_Cap_ID anywhere is a drilldown link. Consistent teal-on-hover, no underline.

 6

#### Failure is a designed state

 Degraded-mode banners are persistent and amber. Empty states carry a recovery CTA, never a bare 'No data'.

 7

#### Mechanical governance

 Every state-mutating button previews what will change. Apply and confirm are teal; revert and reject are orange.

## The 27 pages, plus access

 Nine functional groups, A through I, plus the login screen and the header Settings surface. Each entry leads with the page’s objective - the one job it exists to do - followed by what it contains. A short note up top explains the is_admin access model.
 Roles & access Access is a single is_admin flag, not full role-based access control (RBAC beyond is_admin is out of scope for v1). Every authenticated Zennify user is a power user with full read and the day-to-day approval actions. Admins additionally get: the knowledge graph (which lives in the admin area), catalogue upload & source-registry edits (AG-7), version revert (AG-5), on-demand digest generation (AG-6), and the schema-mapping step that provisions each catalogue version’s database. These admin actions live on existing pages plus the Settings and schema-mapping surfaces.
 A · Explore 4 pages
 A1 Mission control
 Answer ‘is the system healthy?’ and ‘what needs me right now?’ in one glance.
 Four pillar tiles (subcap count, active/inactive, decay candidates, % completeness), a story-concentration heatmap of the most-delivered subcaps by quarter with productized-offering coverage, source-health widget, open suggestion and flag counts, recent reasoning-chain runs, quick links. Empty state routes a first user to Settings to upload a catalogue.

 A2 Capability explorer
 Drill down interactively from pillar to a single subcap, then open its deep dive for use cases and user stories.
 Interactive column drilldown Pillar → Category → L1 cluster → Sub_Cap_ID (each selection filters the next), debounced search-as-you-type, lens-stack filters, right preview pane with Open deep dive CTA. The deep dive it opens is where use cases (tab 5) and user stories are read.

 A3 Subcap deep dive
 Show everything known about one subcap on one scrollable page.
 Stacked sections: identity + status toggle, completeness, M1–M5 maturity, personas, L3/L4, use cases, user stories, themes, offerings, subvertical applicability, SOW/news/vendor signals, benchmarks, lifecycle, reasoning chains, Run AI audit.

 A4 Value chain atlas
 Show the 8 universal value-chain clusters and where the catalogue sits in each, by subvertical.
 Eight MECE clusters as a horizontal pipeline (per-cluster subcap counts) plus radial wheel. Subvertical filter reveals SV-specific stage names from tab 21; expand a cluster → stages → mapped subcaps → deep dive.

 B · Catalogue tools 4 pages
 B1 Platform catalog
 Browse the technology platforms the catalogue is built on, grouped by vendor.
 ~52 L3 platforms (deduped across pillars). Per-platform card with ID, category, subcap and L4 counts, reference URL. Click → addressing subcaps, or jump to vendor intelligence.

 B2 Use case explorer
 Browse the use-case layer by archetype, and drop into the owning subcap.
 ~3,000+ use cases across 22 archetype tags in 5 families. Family tabs, tag cards with counts and examples, click a use case → parent subcap deep dive.

 B3 Maturity heatmap
 Spot where maturity descriptors are missing or thin across the catalogue.
 Subcap × M1–M5 (4,255 cells) filterable by pillar, lifecycle, persona, tier. Aggregation switch (L1×M), cell-hover descriptor preview, click → deep-dive maturity section.

 B4 Knowledge graph
 Reveal the structural relationships the flat catalogue hides. Admin-only.
 Two-layer graph in the admin area. Layer selector (deterministic / +AI-proposed / full), node-kind filter, centrality and community panels, shortest path, impact analysis, Cypher box, mini-map, synthesis-findings tab, right detail drawer. Only admins review and commit AI-proposed edges.

 C · Project validation 3 pages
 C1 SOW library
 Use AI to read each ingested SOW and match it to the subcaps it actually delivered.
 SOWs by status with a redaction-confirmation banner; AI matches SOW text to subcaps using a model tuned on the catalogue ontology (subcap names, descriptions, L4 features and the story-to-subcap pairs), each match carrying a claim label, confidence and reasoning link; per-SOW mention list with excerpts. Sort by ingestion date, match count or confidence.

 C2 Story library
 Browse and filter the canonical Jira story corpus.
 14,406 real client stories (the canonical Jira catalog) with server-side pagination. Synthetic and use-case-derived stories are excluded by default and only appear, clearly labelled, when a news or trend signal flags the subcap as emergent. Filter by pillar / subcap / quality threshold / match confidence, with a per-story quality and reasoning breakdown.

 C3 Project-subcap trace
 See everything that has ever touched a subcap, in time order.
 Pick a subcap: chronological timeline mixing SOW, story, benchmark, news, vendor, lifecycle and suggestion events, each with excerpt, confidence and reasoning link. CSV export.

 D · Public intelligence 4 pages
 D1 News watch
 Turn raw public news into subcap-scored signal.
 Impact-segmented news filtered by source, tier, FS relevance, date, affected pillar, subvertical and magnitude. Per-item card with top-3 affected subcaps + magnitude chips, reasoning link, Run consultant loop.

 D2 Trends monitor
 Watch the trends being staged for the monthly consultant cycle.
 Aggregated trends with affected subcaps, evidence count, time window, status (staged / promoted / consumed). Drill into the evidence cluster. A trend here is also what can promote an otherwise-excluded synthetic story into view, clearly labelled as provisional, when it points to a subcap as emergent.

 D3 AI suggestions
 Let a pillar lead confidently apply or reject AI-proposed catalogue edits.
 Monthly-cycle suggestions filtered by kind / status / gate / pillar / breaking. KPI bar, suggestion cards with rationale, evidence trail, reasoning link, Apply / Reject. Breaking → Review (COMMIT tier).

 D4 Benchmarks studio
 Ground catalogue claims in defensible external benchmarks.
 Distribution charts with bootstrap CI band, methodology and sources panel, subcap mappings, adversary verdict chip (BENCHMARK / INDICATIVE / EXPLORATORY), QoQ toggle, coverage-gap warnings.

 E · Strategic synthesis 1 page
 E1 Quarterly strategic digest
 Consolidate a quarter of signal into board-ready priorities per subvertical.
 Quarter selector, Opus-authored executive summary, cross-SV themes, 9 collapsible subvertical sections with 3–5 priority cards (narrative, evidence trail, recommended actions, adversarial review, claim labels). QoQ delta toggle, Export PPTX (signed, admin-gated generate), per-priority comments.

 F · Lifecycle & competition 3 pages
 F1 Lifecycle manager
 Trace subcaps through delivery and turn that history into productization.
 Ranks the most-delivered subcaps (from SOWs and stories), maps each to a productized offering, flags high-delivery subcaps lacking an offering, and bundles those into an AI-named, AI-scoped proposed new offering with an evidence basis and reasoning link. Promote to a staged suggestion. Replaces the old market-signal lifecycle Kanban, which overlapped the SOW library and client journey atlas.

 F2 Vendor intelligence
 Understand the platform vendors Zennify builds on and the developments that should move the catalogue.
 Vendor profiles (90-day developments, dominant subcaps, lost-to flags, peer adoption %); a developments feed classified into 8 competitive-event types (product_launch, partnership, deprecation, pricing_change, executive_move, security_incident, regulatory_action, case_study) each mapped to affected subcaps with a claim label; an evidence-driven vendor×subcap impact heatmap; per-vendor consultant loop.

 F3 Client journey atlas
 See a unified, entity-resolved view of each client and hand a packet to the DMA team.
 Client list, per-client timeline with 9+ event types, KPI strip (subcaps touched, vendor stack, last DMA score), Export DMA packet as validated signed JSON.

 G · Versioning & QA 5 pages
 G1 Version timeline
 Make every catalogue snapshot inspectable and reversible.
 All snapshots with label, summary, creator, timestamp, per-pillar subcap counts and current flag. Cascade-aware breaking-change summary, admin-gated revert.

 G2 Diff viewer
 Show exactly what changed between two catalogue versions.
 Field-by-field: added / removed / modified subcaps, pillar and category deltas. Explain diff (Gemini Pro narrative), export PDF (signed).

 G3 Change flags inbox
 Surface the anomalies a pillar lead must resolve before trusting the queue.
 Flags filtered by severity / kind / target / age. Per-flag card with resolution form; acknowledge / resolve / dismiss; purpose-built AI-proposed-edge and entity-conflict workflows.

 G4 Validation gates log
 Give transparency into how the 8 deterministic gates are running.
 Per-gate pass / warn / fail distribution over 90 days, recent runs with chain links, gate-failure remediation hints, cost KPIs.

 G5 QA & audit dashboard
 Prove the system is accurate, on-budget and auditable.
 Gate pass-rate per week, hallucination and citation checks, daily / monthly LLM spend against envelope, reasoning-chain volume, source-pull failures, open findings, eval scores per golden dataset.

 H · Reasoning & RAG 2 pages
 H1 Reasoning chain viewer
 Let anyone audit, in plain language, exactly how an AI output was reached - the core trust surface.
 Full-screen modal that reads top to bottom with no opaque codes: a titled question, the ordered steps the model took, the evidence with readable source descriptions (who said it, how trustworthy, how recent), any contradictions, the conclusion, an adversarial self-review, and the checks it ran shown as plain pass / needs-review results, plus a note that the run is reproducible.

 H2 AI chat
 Answer catalogue questions with grounded, citable evidence.
 Conversation list, thread with inline citations and a reasoning backlink per turn, source-hover previews, visible cost meter, context panel, Save / Export PDF.

 I · Sandbox 1 page
 I1 What-if simulator
 Let a lead test a catalogue change safely before proposing it.
 Read-only sandbox. Action editor across 7 kinds (incl. workbook-native toggle with full XLOOKUP cascade), simulate, lifecycle and adoption delta panels, transition list, affected-subcap count, KG orphans, benchmark recompute preview, Promote to suggestion.

 Access & administration 3 surfaces
 AUTH Login
 Gate the workbench to Zennify staff only.
 The only pre-auth surface. Firebase Auth + Google sign-in, restricted to @zennify.com; the domain check fails closed, so non-staff never see product surface. Client-facing usage is explicitly out of scope.

 SCHEMA Schema mapping
 Give an admin control over how each catalogue version maps to the canonical model, provision its persistent relational database, and carry the canonical story mappings forward.
 Opens on upload. Auto-maps each source field to a canonical field with a confidence; the admin confirms or overrides every mapping, adds custom fields, defines relations for that version, and can cascade compatible relations to other versions. Applying writes the backend schema and provisions the persistent relational database for that catalogue. It also carries the canonical Jira story mappings forward from v5: each is similarity-confirmed, with confirmed links maintained and borderline ones routed to review. Stored per version and re-runnable. Admin-only.

 CFG Settings
 Personal preferences for everyone; catalogue and source administration for admins.
 Header-accessible from every page. All users: theme, default persona, default lens. Admin section (gated by the is_admin flag, visually distinct and announced as admin-only): upload pillar workbooks of any version (which launches schema mapping) or configure a Drive folder, edit the ingestion-source registry, trigger manual pulls, set the cost envelope, view secret-rotation status. First successful ingest persists for every user, no re-upload.

 WCAG 2.1 AA is the floor. Dark mode ships in v1 and every surface is verified in both themes.
 Dark mode is in v1 scope. Both themes pass WCAG AA on all surfaces. Colour is never the only signal, per WCAG 1.4.1.

#### Flash prevention
 A synchronous inline script reads the stored theme and sets data-theme before first paint, so dark users never see a light flash.

#### Keyboard + focus
 Focus traps in modals, Esc to close, focus returns to the trigger. Drilldowns move focus to the page H1. A 2px teal focus ring on every interactive element.

#### Screen readers
 Degraded banners use role=alert; status uses aria-live polite. Cost meter is a labelled progressbar. Skip-to-content link on every page.
