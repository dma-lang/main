 Section 1

## Overview and how to read
 This is the delivery team’s build plan. It turns the PRD, TRD, App Flow and Backend Schema into sequenced tasks across three phases, each with an owner role, a measurable done criterion, an effort estimate in engineer-days, and a dependency chain so nothing starts before its inputs are merged.

### 1.1 How to read it
 Role codes: BE backend, FE frontend, ML machine learning, OPS platform. Effort is in engineer-days (1d = one focused day). Dependencies are task IDs; never start a task before its dependencies are merged. Tasks reference the requirement they satisfy ( FR-n ) and the document section that specifies them.

### 1.2 Team assumption
 The plan assumes a small team working parallel tracks (one backend-primary, one frontend-primary, ML tasks shared or deferred where flagged). A single engineer can run the same sequence; each phase simply takes longer. The plan is greenfield: it builds the foundation first rather than modifying an existing codebase.
 The critical path runs through Phase 1
 Until a catalogue version is provisioned (P1-06) and the canonical stories are carried forward (P1-09), no downstream surface has data. Phase 1 is the gating work; Phases 2 and 3 layer onto it and are each shippable on their own.

 Section 2

## Build sequence at a glance
 The work is three phases plus a cross-cutting track, roughly 105 engineer-days in total, which is about 10 to 12 calendar weeks for a two-to-three person team running backend, frontend and ML in parallel.
 | Phase | Scope | Eng-days | Requirements

 | Phase 1 foundation | Service, auth, schema-mapping engine, per-version provisioning, sheet relationships and cascade, canonical Jira ingest and carry-forward, relational read surfaces, versioning, dark mode. | ~41 | FR-1 to FR-6, FR-22
 | Phase 2 evidence & reasoning | Evidence ingestion, ERS and reasoning chains, the eight gates, suggestion lifecycle, vendor and benchmarks, lifecycle manager, monthly cycle and digest, grounded chat, signed exports. | ~40 | FR-7 to FR-17
 | Phase 3 live & automation | Continuous Jira, client journey and DMA packet, admin knowledge graph, what-if sandbox. | ~14 | FR-18 to FR-21
 | Cross-cutting | Eval harness, SLO alerts, cost attribution and degraded mode, audit and security. Runs across all phases. | ~10 | NFRs

### 2.1 Parallel tracks and merge points
 Backend owns ingestion, provisioning, the gates and the API; frontend owns the shell, state and surfaces; ML owns embeddings, matching and the eval harness. The tracks merge at three recurring points: schema model changes (frontend consumes generated API types), new endpoints (frontend consumes them), and the design tokens (owned by frontend, consumed everywhere).

 Section 7

## Release gates
 Each phase has an explicit exit gate tied to the PRD acceptance criteria. A phase is not done until every item passes.

### Phase 1 foundation

- Both v5 and v7 provision a persistent relational database that is durable for all users with no re-upload.
- 851 subcaps and the 14,406 real Jira stories are queryable; synthetic stories are excluded by default.
- The sheet relationships are captured and drive both the foreign keys and the subcap toggle cascade preview.
- v5 story mappings carry forward to v7 with similarity confirmed; borderline matches are queued, none dropped.
- Dark mode and the six-object filter state work across navigation and deep links.

### Phase 2 evidence & reasoning

- Evidence maps to subcaps with an ERS and an explainable reasoning chain.
- Every AI proposal runs the eight gates; failures route to change flags; an apply only commits after passing.
- The monthly cycle stages suggestions and the quarterly digest publishes with adversarial verdicts.
- Catalogue-grounded chat passes the fixed query set, with citations and a reasoning backlink per turn.

### Phase 3 live & automation

- New Jira stories ingest continuously and match, with low-confidence routed to review.
- The DMA packet exports and validates and is signed.
- Knowledge-graph Layer B proposals are gated and human-approved; nothing auto-merges.
- The what-if sandbox previews a cascade without committing anything.

 Section 8

## Sequencing notes
 Sequencing notes that keep the critical path clear; none of these blocks the Phase 1 foundation.
 | Note | Guidance

 | Provisioning is the gate | Sequence P1-04 to P1-06 first and protect it; every later task depends on a provisioned version.
 | Relationships before surfaces | Land the schema-mapping relations (P1-05) and the cascade engine (P1-07) before the read surfaces, so the surfaces and the toggle preview are correct from the start.
 | Carry-forward calibration | Run P1-09 on the real v5-to-v7 data early; tune the 0.86 / 0.70 thresholds from the observed confirmed/review split before relying on them.
 | ML can be deferred, not skipped | If no ML engineer is available, ship Phase 1 with the baseline embedding and defer the fine-tune (X-01); matching quality improves later without reshaping the data.
 | Models may be unavailable | Build P1-14 hermetic mode and X-03 degraded routing early so progress never blocks on provider credentials.
