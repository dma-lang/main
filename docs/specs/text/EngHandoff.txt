## 01 Product overview & users

 An internal, single-tenant web application for ~10–30 expert users (Zennify pillar leads, consultants, data architects). It keeps the four-pillar capability catalogue provably current by ingesting two corpora — the four-pillar capability workbooks (across versions) and the canonical Jira story corpus — reconciling versions into one canonical model, provisioning a persistent relational database per version via an admin-controlled mapping step, and serving trust-first read & reasoning surfaces over that data.

 Non-negotiable product principle — trust-first. Every AI output carries four pieces of metadata: a claim label (FACT · INFERENCE · HYPOTHESIS · CEILING_ESTIMATE), a source tier (T1–T5), an evidence reliability score (ERS) , and a reasoning chain the user can open. These are not decoration — they gate commits (§9).

 | Catalogue scale (v7) | Value | Notes

 | Subcapabilities | 851 | P1 205 · P2 292 · P3 164 · P4 190

 | Pillars → categories → L1 clusters → subcaps | 4 → 16 → ~120 → 851 | The real taxonomy hierarchy used by the navigator

 | Canonical Jira stories | 14,406 | Real client stories; 87 mapped subcaps; synthetic excluded by default

 | L3 platforms | ~135 | Deduplicated entities, vendor-tagged

 | Productized offerings | 13 | Shared across pillars

## 02 System architecture

 A single Cloud Run service hosts the FastAPI backend and the built React SPA behind one container. Asynchronous work is split into Cloud Run Jobs . The system of record is a relational database; object storage holds raw uploads, snapshots and signed exports.

 React SPA Vite · router v6 · Zustand · TanStack Query

 Cloud Run FastAPI + static SPA, one container

 Cloud SQL PostgreSQL 16 · pgvector · full-text

 Jobs / GCS ingest, carry-forward, digest · buckets

 Why relational. The catalogue is inherently relational (subcaps↔platforms, stories↔subcaps, offerings↔subcaps). The mapping studio defines those relations as foreign keys; the same Postgres engine serves hybrid retrieval through pgvector (dense) and tsvector (lexical) alongside ordinary SQL filters — one store for structure, search and provenance.

## 04 The critical path: bringing data online

 The defining flow — and Phase 1's entire reason to exist — is how raw workbooks become queryable, versioned, relational data that delivery evidence is attached to. No downstream surface has data until a version is provisioned and stories are carried forward.

 Upload Admin drops pillar workbooks or points at Drive

 Map Auto-map · confirm · relations · cascade

 Provision Per-version schema + FKs + load rows

 Carry-forward Jira mappings moved to this version

 Serve Read · reason · suggest · export

 This exact flow is the prototype's first-run onboarding (Login → Upload → Schema mapping → Carry-forward → Mission control) and must be the production entry experience for an admin's first session on a fresh version.

## 05 Data model — control plane + per-version data plane

 The database is organised as a shared control plane and a per-version data plane . The full DDL ships in schema.sql / the Backend Schema document; this is the orientation an engineer needs before reading it.

### 5.1 Control plane (shared) — control.*

 Governs identity, versions, ingest runs, the mapping configuration, the canonical story corpus + carry-forward, and all evidence, reasoning, suggestions, gates and audit. Does not change shape when a new catalogue version arrives.

 | Table | Key columns | Purpose

 | users | uid PK · email · is_admin · preferences jsonb | Identity + the single authz flag + per-user prefs (theme, lens, persona)

 | catalogue_version | version_id PK · label · status · schema_name · created_at | Registry of versions; schema_name points at the data-plane namespace

 | ingest_run | run_id PK · version_id FK · source · status · stats jsonb | Every ingestion attempt and its outcome

 | catalogue_sheet | version_id · sheet_name · sheet_role · maps_to_entity | Which workbook sheet populates which canonical entity

 | relation_def | from_entity · from_field · rel_type · to_entity · to_field · card · via_sheet · is_cascade · cascade_kind | The relation registry — provisioning turns these into FKs (one/many-to-one) and link tables (many-to-many); the cascade engine reads them

 | story | story_key PK · subcap_id · scores · confidence · is_synthetic · … | Canonical Jira corpus (14,406), control-plane so it survives version changes

 | carry_forward | story_key · from_version · to_version · status · similarity | Results of moving v5 story→subcap mappings to the active version

 Enums to honour exactly (from schema.sql ): mapping_status = confirmed · review · custom · ignored · unmapped; carry_status = confirmed · review · unmapped; relation_type = belongs_to · addresses · delivered_by · maps_to_offering · uses_platform · tagged_theme · has_persona · has_usecase · custom; evidence_kind = sow_chunk · news · vendor_event · benchmark · regulatory · catalogue; claim labels = FACT · INFERENCE · HYPOTHESIS · CEILING_ESTIMATE.

### 5.2 Data plane (per version) — cat_v5.* , cat_v7.*

 Each version owns a Postgres schema namespace holding the canonical entities for that version: subcap , category , capability , l4_feature , persona , use_case , maturity_descriptor , and the deduplicated entities ( offering , data_product , l3_platform , theme , agent ) with their link tables ( subcap_platform , subcap_persona , offering_subcap , …). The mapping studio generates this schema; the app reads the active version by name .

#### Why schema-per-version (not a version column)

 Physical isolation: re-provisioning or reverting a version never risks another version's rows, and one version's admin-defined relations can't constrain another. Cross-version questions (lineage, diff, carry-forward) are answered by the control plane , which holds the crosswalk ( _R1_Source_Reference ) and the carry-forward results.

 The real v7 relations the prototype's mapping studio shows (and that provisioning realises) — these are the authoritative relation_def rows:

### 5.3 Handling both catalogue versions (v5 & v7)

 The app must serve both the legacy v5 and the current v7 catalogue from the same code, switching by the header version toggle. They are physically separate data-plane schemas ( cat_v5 , cat_v7 ); the control plane reconciles them.

 | Dimension | v5 (legacy) | v7 (active)

 | Workbook shape | 10 sheets/pillar, inconsistent naming | 25 canonical tabs/pillar

 | Subcaps | ~840 (P1 199 · P2 292 · P3 162 · P4 187) | 851 (P1 205 · P2 292 · P3 164 · P4 190)

 | FC subvertical | absent | +11 net-new finance-company subcaps

 | Stories | workbook gap-counts only | superseded by canonical Jira export

 | Field map (studio) | Sub-Cap ID , Sub-Capability , Primary Products , Key Features (lower auto-map confidence) | Sub_Cap_ID , Sub_Cap_Name , L3_Platforms_Addressing_SubCap , L4_Features_Available (high confidence)

 | Relations realised | 4 (belongs_to, uses_platform, addresses l4, maps_to_offering) | 9 (adds has_persona, has_usecase, addresses maturity, delivered_by, value-chain)

 Engineering rules: (1) each subcap keeps one identity across versions via the _R1_Source_Reference crosswalk in the control plane; (2) every count, diff and lineage payload is version-qualified (R3); (3) the Diff viewer reads the crosswalk to compute added/removed/modified between any two versions; (4) carry-forward (§7) is what moves the v5-mapped Jira corpus onto v7; (5) reverting a version restores its schema snapshot but never rewinds the evidence-corpora timestamps (the record of "what was known when" is preserved).

## 06 Schema-mapping engine (FR-2)

 What makes the system resilient to however a future catalogue version structures its workbook. Runs on upload, admin-controlled end to end. A new catalogue structure must never require a code release.

 | Step | What happens | Engineering note

 | Detect | Identify source layout, pick parsing adapter | Deterministic; unknown layout → manual mapping, not a failure

 | Auto-map | Propose a canonical field per source field + confidence | Embedding + heuristic match on header names & sampled values

 | Confirm / override | Admin accepts high-confidence, resolves flagged | Every field ends confirmed , custom , or ignored

 | Custom fields | Add columns with no canonical home | Nothing in source silently discarded

 | Relations | Define FK relations for this version from the UI | Writes relation_def rows (see §5.2)

 | Cascade | Propagate compatible relations to other versions | Only where fields exist and are type-compatible

 | Apply / provision | Write schema, create tables + FKs, load rows | Transactional & idempotent ; re-runnable to correct a mapping

 # ingestion/provision.py — provisioning is ONE transactional, idempotent op keyed on version_id
 def bring_version_online(version_id, workbooks):
 layout = detect_layout(workbooks) # "v5" | "v7" | unknown→manual
 proposal = automap_fields(layout, CANONICAL_MODEL) # [(src, canon, confidence, status)]
 mapping = await admin_confirm(version_id, proposal)
 mapping += admin_custom_fields(version_id)
 relations = admin_relations(version_id) + cascade_compatible(...)
 with txn():
 create_schema( f"cat_{version_id}" )
 for e in CANONICAL_MODEL: create_table(e, mapping[e] + custom_cols(e))
 for r in relations: add_foreign_key(r)
 load_rows(workbooks, mapping)
 register_ingest(version_id, status= "provisioned" ) # durable for ALL users (D10)

## 07 Story pipeline & carry-forward (FR-3, FR-4)

 The canonical story source is the Jira Full Story Catalog (14,406 real stories, 87 mapped subcaps) — the analysis set. It supersedes the workbook story tabs and is persisted in the control plane . Synthetic / use-case-derived rows are loaded but excluded from analysis by default ; they surface only when a trend flags a subcap as emergent, then labelled provisional .

#### Carry-forward correlation — deterministic first, semantic second

 Because the catalog was mapped against v5, carry-forward maintains those mappings on the active version: (1) normalise the id, stripping any subvertical suffix (e.g. P3C1.8.CL2 → P3C1.8 ); (2) bridge with the target version's crosswalk; (3) confirm similarity with the catalogue-tuned embeddings.

 Thresholds: ≥ 0.86 → carried automatically · 0.70–0.86 → one-click admin confirm · < 0.70 → unmapped with a nearest-neighbour proposal. Idempotent & re-runnable; results written to the control plane. Ships as carry_forward_subcap_mappings.py .

 One embedding space backs story matching, SOW matching and carry-forward, so a subcap means the same thing whether matched, retrieved, or carried between versions.

### 7.5 SOW enrichment & subcap mapping pipeline to build

 The ingested SOW corpus is raw and not yet mapped to subcaps — this is the work to do. The current export is 132 SOW files → 1,075 extracted scope items with columns: SOW Filename · Opportunity ID · Account Name · Opportunity Name · Scope Item/Task Name · QTY · Scope Description · Assumptions . There is no subcap link yet.

 Ingest + DLP Raw SOW → IAM bucket; redact before any model sees it

 Normalise One row per scope item; clean task name + description

 Embed Scope item → shared catalogue embedding space

 Match Scope item → subcap(s) w/ cosine confidence

 Gate + confirm G1–G8; ≥0.86 auto, 0.70–0.86 review, <0.70 unmapped

 Persist SOW→subcap evidence rows w/ claim label + chain

 | Stage | Detail

 | Source of record | Each scope item becomes an evidence(kind=sow_chunk) row in the control plane, carrying account_name , opportunity_id , sow_filename , qty , assumptions .

 | Matching model | Gemini Pro / Claude Sonnet over the one shared embedding space (§7) so "subcap" means the same as in story-matching & carry-forward. Output: [{subcap_id, confidence, claim_label, rationale}] .

 | Thresholds | Reuse the carry-forward bands: ≥0.86 auto-confirm · 0.70–0.86 → one-click admin confirm in the SOW library · <0.70 → unmapped + nearest-neighbour proposal.

 | Gating | Every proposed SOW→subcap link runs G1–G8 (esp. G5 similarity, G7 citation) before commit; failures route to Change Flags.

 | Downstream use | Confirmed links power the SOW library (matched subcaps + Commit), the Project-subcap trace (SOW events), and the Lifecycle manager (co-delivery → offering opportunities).

#### This is greenfield work, not a re-skin
 The prototype's SOW library shows the target end-state (matched, confirmable, committable). Production must build the enrichment pipeline above to get there from the 1,075 raw scope items. Until it runs, SOW surfaces show "ingesting / unmatched".

## 08 AI / ML subsystem (FR-7…FR-17)

 Grounded and tiered. Answers and matches are retrieved from stored catalogue and evidence, never from model memory . Model strength is matched to leverage.

 | Job | Model tier | Use

 | Classification / extraction | Gemini Flash | News tagging, event typing, story field extraction — high volume, low cost

 | Matching / reasoning loop | Gemini Pro / Claude Sonnet | SOW→subcap & story→subcap matching, impact scoring, reasoning steps

 | Synthesis / adversarial review | Claude Opus | Quarterly digest, benchmark adversary verdict — bounded volume

 | Embeddings | Catalogue-tuned | Retrieval, matching, carry-forward — one shared space

 Retrieval is hybrid in one engine: lexical ( tsvector +GIN) + dense ( pgvector HNSW) + structured SQL filters (pillar, subvertical, claim label, source tier). The catalogue ontology itself is part of the retrieval corpus so chat is catalogue-grounded (FR-15).

 Cost control: every model call is tagged (operation, pillar, subvertical, subcap, batch) for exact cost-by-feature. Alert at 80% of the monthly envelope; throttle non-essential ops at 90%; auto-route to a fallback model with a degraded-mode banner rather than failing. Frontend note the cost meter and degraded banner exist in the prototype and are admin-only .

## 09 Eight validation gates (G1–G8)

 Deterministic gates that sit between any AI proposal and any commit . They are code, not prompts — reproducible and auditable. A proposal failing any gate routes to the Change Flags inbox with the failing gate named; it is never auto-applied. Each run persists a validation_gate_run linked to its reasoning chain.

 | Gate | Name | Check

 | G1 | Identity & schema | Target subcap resolves in the active version; payload validates against the canonical model

 | G2 | Evidence sufficiency | Claim has the minimum evidence its label requires; bare assertions rejected

 | G3 | Source-tier floor | Every cited source meets the minimum tier; a commit needs stronger sourcing than a note

 | G4 | Claim-label consistency | Asserted label matches evidence strength; no FACT on a single weak source

 | G5 | Similarity / grounding | Matches & carry-forward meet the cosine floor; chat answers overlap retrieved context

 | G6 | Contradiction check | A proposal conflicting with committed evidence is flagged contradicted_evidence

 | G7 | Citation verification | Every citation resolves to a stored, retrievable chunk; fabricated citations fail

 | G8 | Budget & rate | Operation within cost envelope & rate limits; degraded-mode routing applied

## 10 Knowledge graph (admin-only, D18)

#### Layer A — deterministic
 A projection of committed structure: foreign keys and link tables become edges. Always trustworthy.
 BELONGS_TO ADDRESSED_BY DELIVERED_BY MAPS_TO_OFFERING USES_PLATFORM TAGGED_THEME HAS_PERSONA HAS_USECASE EVIDENCED_BY

#### Layer B — AI-proposed, gated
 Edges the structure doesn't state outright. Must pass G1–G8 and human approval in Change Flags before commit. Limited to SEMANTICALLY_SIMILAR (cosine ≥ 0.85) and SHARES_FEATURE . Cross-pillar dependency edges are out of scope (D16). No AI edge is ever written live — the proposer emits a pending_edge .

## 14 State, caching & real-time

- Server cache : TanStack Query keyed by [version, resource, filters] . The active version is part of every catalogue key so a version switch refetches cleanly.

- Client state : Zustand for ephemeral UI (open tab, drawer, selected node). Filter state is URL-serialized (pillar, sv, lens, page, search) for deep links — mirror the prototype's hash params.

- Heavy lists (stories 14k, use cases) are server-paginated & server-filtered — never ship the full set to the client (SLO §15).

- Long jobs (provision, carry-forward, monthly cycle, digest) are async Cloud Run Jobs — the UI polls ingest_run.status / job status and shows progress, exactly like the onboarding stepper.

- Chat is streamed; digest/what-if may be long — show the reasoning/progress affordances the prototype already has.

## 16 Environments & hermetic dev

 Three environments — dev (hermetic LLM mode: the full stack runs with no provider credentials , model stack stubbed deterministically), staging (real providers, synthetic admin), production . The same container image is promoted; only config/secrets differ. Provider availability is a config switch, not an architectural dependency. Tip the delivered prototype is effectively the hermetic-dev frontend — deterministic data, no provider calls — so it doubles as the dev-mode UI.
