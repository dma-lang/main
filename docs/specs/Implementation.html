Section 1

How to read this build manual

This is the build manual for the Capability Intelligence Agent. It is written so an engineer (or a coding agent) can build the whole system without guessing and without doing the same work twice. Read section 2 first: it specifies the shared foundations every page sits on, including the intelligence layers (automap, enrichment, matching, reasoning, gating) that keep the catalogue current and resilient. Then build pages from sections 3 to 13; each page only adds its own surface-specific work and names the foundations it reuses, so there are no overlaps.

Every surface uses the same fixed template, so nothing is implicit:

1.2 Conventions

  • Roles: BE backend, FE frontend, ML machine learning, OPS platform.
  • Version-awareness: the active catalogue version (v7 today, v5 legacy) is in the header context and part of every catalogue read and every TanStack Query key; counts, diffs and lineage are version-qualified.
  • Trust metadata is not optional: wherever a surface shows an AI-derived value it shows the claim label, source tier, ERS and a reasoning-chain link. These gate commits (F8).
  • Hermetic dev: the delivered prototype is the dev-mode frontend (deterministic data, no provider calls); build against it and swap the in-memory dataset for the endpoints surface by surface.

Section 2

Shared foundations (build once, incl. the intelligence layers)

Build these fifteen foundations before the pages. Every surface reuses them and never reimplements them - this is what keeps the build free of overlaps. F4, F5 and F13 carry the intelligence this revision adds (automap that survives renamed columns, the two-fold story pipeline with enrichment, the learning loops that compound); F14 and sections 15-16 carry the resilience (self-healing at every stage and safe Cloud Run deployment and migrations); F15 makes the backend schema’s relationships explicit and proposes new ones; section 17 covers the Gemini model strategy.

F1 · Service skeleton & environments (OPS, BE)

One Cloud Run service hosts the FastAPI backend and the built React SPA in a single container. Three environments - dev (hermetic: the full stack runs with no provider credentials, all model calls stubbed deterministically), staging (real providers, synthetic admin) and production - promote the same image; only config and secrets differ. Provider availability is a config switch, never an architectural dependency.

Build steps
  1. Scaffold the FastAPI app with a routers package (one router per group in F9); serve the Vite React build as static files from the same container; the app reads PORT and listens on 0.0.0.0 (never 127.0.0.1).
  2. Build the image for linux/amd64 (docker buildx) - an arm/Mac image is the most common "container failed to start"; verify the image runs locally before deploy.
  3. Expose a cheap /healthz (readiness) and a deeper /livez (liveness); configure a Cloud Run startup probe so traffic only routes when ready, and a liveness probe so a hung instance is restarted (section 16).
  4. Handle SIGTERM: stop accepting new work, drain in-flight requests, close the DB pool, and exit within the 10-second window before SIGKILL.
  5. Set --min-instances 2 on the critical path (removes the rolling-deploy cold-start gap) with startup CPU boost; set --max-instances as downstream protection sized to the Cloud SQL connection budget, not peak traffic.
  6. Wire Secret Manager (mounted, never baked into the image or plain env); run as a least-privilege service account; expose LLM_MODE (hermetic | live) with a deterministic stub model stack for dev.
  7. Deploy by revision with a canary then promote, and keep auto-rollback to the last healthy revision; CI builds the image once and promotes it across environments (section 16).

Done when. the amd64 image builds and runs locally, /healthz gates traffic, SIGTERM drains cleanly within 10s, a canary revision can be promoted or rolled back, and the app runs fully in hermetic mode with no provider credentials.

F3 · Control-plane schema & migrations (BE)

All control.* tables and enums from schema.sql created by versioned migrations - identity, versions, ingest runs, mapping configuration, the canonical story corpus and carry-forward bridge, evidence, reasoning, suggestions, gates, governance, SOW, intelligence and audit. The control plane does not change shape when a new catalogue version arrives.

Build steps
  1. Adopt the control-plane DDL from schema.sql as the Alembic baseline; enable pgcrypto and pgvector; stamp an already-provisioned database to baseline rather than re-running create.
  2. Seed the enums exactly (mapping_status, carry_status, relation_type, evidence_kind, claim_label, catalogue_impact, source_type, offering_tier, data_product_category).
  3. Run migrations as a one-shot Cloud Run Job, never on app startup, and only promote a new service revision after the Job completes (section 16); the app never migrates itself.
  4. Guard the runner (script below) with a Postgres advisory lock on a direct connection (not the transaction pooler), a lock_timeout, and a version-check-first skip so a re-run at head is a no-op.
  5. Keep migrations idempotent and forward-safe: state lives in alembic_version in the database (not the image or Terraform state), so image rebuilds and repeated terraform apply are no-ops at head; use expand/contract for zero-downtime and CREATE INDEX CONCURRENTLY outside a transaction.
  6. Never autogenerate-and-apply in production; review every revision; run migrations against an ephemeral Postgres 16 in CI, and test the downgrade.
app/migrate.pypython
# app/migrate.py - safe, idempotent migrations for an autoscaled service.
# Run as a ONE-SHOT JOB (never on app startup - autoscaled instances would race the migration).
# Use a DIRECT DB connection, NOT the transaction pooler: pg advisory locks are session-scoped and a
# transaction pooler (PgBouncer txn mode) releases the connection between statements, hanging the lock.
import sqlalchemy as sa
from alembic import command
from alembic.config import Config

LOCK_KEY = 0x0C1A_M1   # any stable 64-bit key for this app

def run():
    engine = sa.create_engine(DIRECT_URL, pool_pre_ping=True)     # direct, not via the pooler
    with engine.connect() as cx:
        if at_head(cx):                                           # cheap skip if already at head
            log("schema at head; nothing to do"); return          # -> Job retries / re-apply are no-ops
        cx.execute(sa.text("SET lock_timeout = '30s'"))           # bounded: a stuck holder can't hang deploys
        if not cx.execute(sa.text("SELECT pg_try_advisory_lock(:k)"), {"k": LOCK_KEY}).scalar():
            log("another runner holds the migration lock; exiting 0"); return
        try:
            command.upgrade(Config("alembic.ini"), "head")        # Postgres transactional DDL: a failure rolls back
        finally:
            cx.execute(sa.text("SELECT pg_advisory_unlock(:k)"), {"k": LOCK_KEY})

# Resilience properties:
#  - State lives in alembic_version in the DB, NOT in the image or Terraform state -> image rebuilds are safe.
#  - upgrade head at head is a no-op -> Job retries and repeated `terraform apply` are safe (idempotent).
#  - Expand/contract migrations (add nullable -> backfill -> later drop) keep old + new revisions both working.
#  - CREATE INDEX CONCURRENTLY runs OUTSIDE a transaction (op.create_index(..., postgresql_concurrently=True)).

Done when. the control plane migrates cleanly onto empty Postgres 16, a concurrent or repeated run is a safe no-op, a half-applied migration rolls back (transactional DDL), and an image rebuild or re-apply does not duplicate or lose schema state.

F4 · Schema-mapping engine with an automap intelligence layer + per-version provisioning (BE, ML)

The resilience core (FR-2) and intelligence layer L1. On upload it detects the workbook layout and auto-maps source fields to the canonical model even when column names have changed, because it scores four independent signals and learns from every admin decision. An admin confirms or overrides and adds custom fields; the version’s relations are captured; on apply the data-plane schema (cat_<version>) is provisioned: tables, FKs from relation_def, bulk-loaded rows. A new catalogue structure must never need a code release.

Build steps
  1. Write parsing adapters for v5 (10 sheets/pillar) and v7 (25 tabs/pillar); an unknown layout routes to manual mapping, not failure.
  2. Build the automap intelligence layer (script below): for each source column, score (a) header lexical similarity with a synonym + abbreviation dictionary, (b) header embedding similarity, (c) a value fingerprint that classifies the column by the shape and content of its sampled values (so a renamed column is still recognised), and (d) a learned-memory prior from past confirmations; combine into a confidence and a status (confirmed / review / custom).
  3. Implement the learning loop: every admin confirm or override updates the memory, the synonym set and the canonical embedding centroid, so the next version or tenant maps the renamed column automatically.
  4. Expose admin endpoints /automap, /confirm, /relations (CRUD on relation_def), /apply; persist mappings to source_field_mapping and catalogue_sheet.
  5. Implement the canonical cell parsers (R1): L3 cell [L3-ID] Name (count: N) split + vendor-from-prefix; same discipline for persona, use-case and story-ref cells.
  6. Implement bring_version_online(version_id, workbooks) as one idempotent transaction: create schema, create tables (canonical + custom columns), add FKs, load rows, register ingest_run = provisioned.
ingestion/automap.pypython
# ingestion/automap.py - multi-signal field auto-mapping with a learning memory.
# Resilient to renamed columns: when the HEADER changes, the VALUES still reveal the field.

CANONICAL = ["subcap_id","name","category","cluster","tier","platforms","features",
             "personas","use_cases","maturity","status","story_refs","offering_refs"]

def automap_fields(sheet, canonical=CANONICAL, memory=Memory()):
    proposals = []
    for col in sheet.columns:
        header = normalise(col.header)              # lowercase, strip punctuation, expand abbreviations
        sample = col.sample_values(n=50)            # for content fingerprinting (name-independent)
        scores = {}
        for canon in canonical:
            s_lex = lexical_sim(header, SYNONYMS[canon])         # token + synonym + abbrev match
            s_emb = cosine(embed(col.header), CANON_EMBED[canon])# semantic header match (Primary Products ~ L3_Platforms)
            s_val = value_fingerprint(sample, canon)             # data-shape / content classifier
            s_mem = memory.prior(col.header, canon)              # learned from past confirmations
            scores[canon] = 0.30*s_lex + 0.30*s_val + 0.25*s_emb + 0.15*s_mem
        canon, conf = argmax(scores)
        status = ("confirmed" if conf >= CFG.auto      # 0.86 (config, not code)
                  else "review" if conf >= CFG.review  # 0.70
                  else "custom")                       # unknown -> custom column; never silently dropped
        proposals.append(Mapping(col.header, canon, round(conf,3), status,
                                 signals=scores, evidence=sample[:3]))
    return proposals

def value_fingerprint(sample, canon):
    # The content classifier is what makes automap survive a rename - it reads VALUES, not the name.
    if canon == "subcap_id": return frac(sample, regex=r"^P\dC\d")        # P3C1.8 ...
    if canon == "platforms": return frac(sample, contains="[L3-")          # [L3-SF-...]  (R1 grammar)
    if canon == "personas":  return frac(sample, looks_like=role_list)     # semicolon-delimited roles
    if canon == "maturity":  return frac(sample, matches=M1_M5_prose)      # five graded descriptors
    if canon == "tier":      return frac(sample, in_set={"T1","T2","T3"})
    return value_embed_class(sample, canon)                                # else: a value-embedding classifier

# Learning loop - the intelligence that compounds.
# Every admin confirm/override updates the memory, the synonym set and the canonical centroid,
# so the NEXT version (or tenant) auto-maps a renamed column correctly without a code change.
def on_admin_decision(col_header, chosen_canon, accepted):
    memory.record(col_header, chosen_canon, accepted)
    if accepted:
        SYNONYMS[chosen_canon].add(normalise(col_header))
        CANON_EMBED[chosen_canon] = update_centroid(CANON_EMBED[chosen_canon], embed(col_header))

Done when. a workbook with renamed columns still auto-maps correctly from value fingerprints and learned memory, the real v7 workbooks provision cat_v7 transactionally, a re-run upserts, and an unknown column becomes a custom column rather than being dropped.

F5 · Canonical story pipeline - seed + continuous enrichment + carry-forward (BE, ML)

The canonical story corpus is two-fold. (1) Seed: the initial export of stories mapped against v5 (14,406 real stories) is loaded as the baseline analysis set and carried forward to the active version. (2) Continuous: the app connects to Jira and monitors new completed stories; each new raw story is enriched into the canonical story shape - the same fields and quality scores the seed stories carry - before it is mapped back to a subcap and persisted. This enrichment is what keeps the whole corpus one consistent shape; intelligence layer L2. Synthetic rows are flagged and excluded from analysis by default.

Build steps
  1. Seed: load the v5-mapped export into control.story (provenance, is_synthetic flag); run seed_native('v5') for native v5 links; run carry_forward('v5','v7') (normalise id, crosswalk, F6 embedding confirm, bands in config) into story_subcap_carry; create the story_catalogue_link view.
  2. Connect to Jira: a Cloud Scheduler poll (or webhook) pulls new completed stories into a raw landing - the continuous monitor.
  3. Enrich (the crucial step): for each raw issue run enrich_story() to derive the canonical fields and quality scores - acceptance-criteria, solution-design and story sub-scores, composite, confidence band, subvertical, target maturity, population, clean summary - so a story ingested today is indistinguishable in shape from a seed story.
  4. Map back to subcaps: match the enriched story to a subcap via the F6 shared embedding space; gate (G1, G5, G7); confirmed auto-link, borderline to review, none dropped.
  5. Persist: write into control.story and story_subcap_carry for the active version; idempotent on story_key so re-runs upsert.
ingestion/story_enrich.pypython
# ingestion/story_enrich.py - raw Jira issue -> canonical story record -> subcap mapping.
# The canonical corpus has ONE shape; every NEW story is enriched into that shape before it is used,
# so a story ingested today is indistinguishable from a seed story exported against v5.

def ingest_jira_story(issue, version):              # issue = raw Jira fields (continuous monitor)
    raw   = extract(issue)                           # key, project, summary, description, AC, labels, components
    story = enrich_story(raw)                        # -> canonical schema (same fields the seed corpus carries)
    match = map_story_to_subcap(story, version)      # embedding match against the ACTIVE version
    if gate_pass(match, gates=["G1","G5","G7"]):     # identity, grounding, citation
        persist_story(story, match, version)         # control.story (+ story_subcap_carry for this version)
    else:
        queue_for_review(story, match)               # borderline -> review; never dropped
    return story, match

def enrich_story(raw):
    # Derive the SAME quality signals the seed stories already carry, so shapes match exactly.
    ac = score_acceptance_criteria(raw.acceptance_criteria)   # 0..3  (Flash/Sonnet rubric)
    sd = score_solution_design(raw.description)               # 0..3
    ss = score_story(raw.summary, raw.description)            # 0..5
    return CanonicalStory(
        story_key   = raw.key, project_key = raw.project,
        summary     = clean(raw.summary),
        sub_scores  = {"ac": ac, "sd": sd, "ss": ss},
        composite   = weighted(ac, sd, ss),
        confidence  = confidence_band(ac, sd, ss),            # HIGH / MEDIUM / LOW
        subvertical = classify_subvertical(raw),              # BK/CL/CIB/FC/CU/WM/RIA/IC/IB
        target_maturity = infer_target_maturity(raw),         # M1..M5
        population  = infer_population(raw),
        source_system = "jira", is_synthetic = False)

def map_story_to_subcap(story, version):
    cands = hybrid_retrieve(story.summary + " " + story.description, version, k=5)   # F6 shared space
    top   = best(cands)
    return Match(subcap=top.subcap_id, similarity=top.score, via="embedding",
                 status=band(top.score),                      # >=0.86 confirmed / 0.70 review / else unmapped
                 chain=reasoning(cands))                      # a reasoning chain is always recorded

Done when. the seed corpus loads and carries to v7; a new Jira story is enriched to the canonical shape, mapped to a subcap, gated and persisted; borderline matches are queued and synthetic rows stay out of analysis by default.

F6 · Embeddings & hybrid retrieval (ML, BE)

Intelligence layer L3. One catalogue-tuned embedding space backs story matching, SOW matching, offering matching, carry-forward and chat, so a subcap means the same thing everywhere. Each subcap carries a tsvector and a vector(768); retrieval is hybrid in one engine: lexical (full-text + GIN), dense (pgvector + HNSW) and structured SQL filters (pillar, subvertical, claim label, source tier).

Build steps
  1. Stand up the embedding service with a deterministic hermetic stub and a live provider behind the F1 switch.
  2. Populate subcap.search and subcap.embedding at provision time; index with GIN and HNSW.
  3. Implement a retrieval function that blends lexical rank and cosine and applies the SQL filters; expose it to chat, all matching and KG Layer B.
  4. Add the eval harness: a golden set of subcap-evidence and subcap-story pairs, tracked as MRR; a drop triggers a contrastive embedding fine-tune.

Done when. a hybrid query returns ranked subcaps within the read SLO and retrieval MRR is tracked on the golden set.

F8 · Validation gates + suggestion lifecycle (BE)

Intelligence layer L5. Eight deterministic gates (G1 identity & schema, G2 evidence sufficiency, G3 source-tier floor, G4 claim-label consistency, G5 similarity / grounding, G6 contradiction, G7 citation verification, G8 budget & rate) sit between any AI proposal and any commit; they are code, not prompts. Suggestions move propose -> gate -> apply or reject; apply re-runs the gates server-side and writes a new snapshot; any failure routes to Change Flags with the failing gate named.

Build steps
  1. Implement each gate as a pure function over (payload, retrieved evidence, config); never inline a gate as a prompt.
  2. Build the gate runner that persists one validation_gate_run per evaluation, linked to its reasoning chain.
  3. Implement the suggestion endpoints; apply runs in a transaction that re-gates, writes the snapshot and an audit row; reject requires a reason.
  4. Route any gate failure to change_flag with the gate id.

Done when. a proposal that fails any gate is never auto-applied and appears in Change Flags named by gate, and an apply is only written after a clean pass with a persisted gate run.

F13 · Intelligence layer & learning loops (ML, BE)

A first-class subsystem that ties the five layers together and the ingest -> enrich -> persist -> map pattern that governs every corpus. It is what makes the app improve with use rather than rot. It owns the model-tier routing, the shared embedding space, the per-source enrichers, and the learning loops that feed confirmations back into the system.

Build steps
  1. Model-tier routing (current Gemini lineup, see section 17): route each job to the cheapest model that meets the bar - gemini-3.1-flash-lite for high-volume classification/extraction (news typing, vendor typing, story field extraction), gemini-3-flash for structured enrichment, gemini-3.5-flash for matching/reasoning with native grounding (story/SOW/news -> subcap), and gemini-3.1-pro-preview (or Claude Opus) for synthesis and adversarial review (digest, benchmark verdict); embeddings use gemini-embedding-2. Models are pinned (never -latest in prod) and every call is cost-tagged (F7, F11).
  2. One shared embedding space (F6): retrieval, story matching, SOW matching, offering matching and carry-forward all resolve the same “subcap”, so a match made in one pipeline is comparable to a match made in another.
  3. The universal pattern: implement ingest -> enrich -> persist -> map once as a reusable job shape; the story (F5), SOW (C1) and evidence (F7) pipelines are instances of it, which is why their scripts look alike.
  4. Learning loops: (1) automap memory + synonym set + centroid update on every admin mapping decision; (2) confirmed/rejected matches become labelled pairs that retrain the matcher and refresh the F6 eval golden set; (3) thresholds (0.86 / 0.70) live in config and are recalibrated from the real v5->v7 run and ongoing precision/recall.
  5. Guardrails: nothing the intelligence layer produces is committed without passing the F8 gates; every output records a reasoning chain (F7).

Done when. each job runs on the right model tier with cost attribution, all pipelines share one embedding space, the three corpora reuse one ingest->enrich->persist->map job shape, and an admin mapping decision measurably improves the next automap.

F14 · Resilience & self-healing patterns (BE, OPS)

A reusable resilience layer that every job, endpoint and pipeline composes, so failures self-correct instead of corrupting state or paging a human. This is how "self-healing at every stage" is actually implemented: the patterns here are imported by ingestion, enrichment, matching, the gates, exports and the cascade engine rather than re-coded per surface. Section 15 maps each stage’s failure modes onto these patterns.

Build steps
  1. Retry with backoff + jitter. Every transient external call (DB, model provider, Drive, Jira) retries with capped exponential backoff and jitter; permanent errors (4xx, schema violations) fail fast rather than retrying.
  2. Idempotency keys. Every job and write carries a natural key (story_key, a scope-item content hash, the ingest_run id) so a retry or a duplicate delivery upserts and never double-writes.
  3. Circuit breaker. Per provider/source; sustained failure trips to the degraded-mode fallback (F11) with a banner, then half-opens to probe recovery - the app stays up on a fallback rather than failing.
  4. Dead-letter queue + poison handling. A message that fails N times lands in the DLQ with its error and payload and is replayable; it never blocks the queue behind it.
  5. Watchdog / reaper. An ingest_run or job stuck in "running" past its lease is detected, marked failed and safely re-enqueued; idempotency makes the re-run safe.
  6. Reconciliation / repair jobs. Scheduled self-checks re-derive the denormalised count views, re-embed any subcap missing a vector, and re-gate rows whose inputs drifted.
  7. Data-integrity self-checks. Validate that every carried_to_subcap reference resolves, detect orphaned evidence/links and dangling KG nodes, and quarantine offenders to a review queue rather than serving bad data.
  8. Checkpointed, drainable jobs. Long jobs checkpoint progress so a SIGTERM mid-batch resumes rather than restarts; graceful shutdown and health/readiness/liveness are wired everywhere (F1).
  9. Auto-rollback. A failed deploy never takes traffic (section 16); a failed apply re-gates and reverts to the last snapshot (F8).
  10. Bounded everything. Timeouts, lock timeouts, max-retries and budget caps (F8 G8) on every path - nothing waits or spends without a ceiling.

Done when. a transient failure retries and succeeds, a duplicate delivery upserts cleanly, a stuck run is reaped and safely retried, a poison message lands in the DLQ without blocking the queue, and an integrity self-check quarantines a dangling reference instead of serving it.

F15 · Relationship model & discovery (intelligent schema linkage) (BE, ML)

The backend schema (control plane + per-version data plane; 71 tables, 20 enums, 2 views in schema.sql) is fully relational, and every relationship is defined in one of five explicit ways. This foundation states them so the implementation is unambiguous, and adds an intelligent layer that proposes relationships the schema does not yet model.

Build steps
  1. Declared relations -> foreign keys. control.relation_def rows (from_entity, to_entity, cardinality, via_sheet, is_cascade, cascade_kind) are the registry; provisioning (F4) turns each into a real FK or link table in cat_<version>, and the cascade engine reads them. The relation_type enum is the closed vocabulary: belongs_to, addresses, delivered_by, maps_to_offering, uses_platform, tagged_theme, has_persona, has_usecase, custom.
  2. Many-to-many link tables. subcap_platform, offering_subcap (+ core_capability, mapping_rationale, maturity_lift), subcap_vcc and the use-case links carry m:m edges with their own attributes.
  3. Cross-version lineage. control.version_crosswalk (from_version / from_subcap / to_version / to_subcap) defines how a v5 subcap maps to v7; the Diff viewer and carry-forward read it.
  4. Story bridge. control.story_subcap_carry (native v5 / carried v7) plus the story_catalogue_link view define the story-to-subcap relationship per version without duplicating the 14,406-row corpus.
  5. Knowledge-graph projection. kg_node / kg_edge project the declared relations as Layer A (deterministic); pending_edge holds Layer-B proposals.
  6. Relationship discovery (the intelligent layer, script below). discover_relationships() scans entity pairs and scores three signals - value overlap (A’s keys hidden in B’s cells = a latent FK), co-occurrence (A and B repeatedly linked via stories or a third table) and embedding affinity (consistently close in the F6 space) - to PROPOSE relationships the schema does not yet model; every candidate is gated (G1, G5, G6) and becomes either a relation_def proposal (admin-confirmed) or a KG Layer-B pending_edge, never a live write.
intelligence/discover_relationships.pypython
# intelligence/discover_relationships.py - propose NEW relationships the schema does not yet model.
# Relationships are normally DECLARED in control.relation_def (-> FKs). This finds undeclared candidates,
# scores three signals, gates them, and routes survivors to human approval. It never writes a live edge.

def discover_relationships(version):
    cands = []
    for a, b in entity_pairs(version):                 # e.g. (subcap, offering), (subcap, l3_platform)
        s_val = key_overlap(a, b)                       # 1) do A's keys appear inside B's cells? (latent FK)
        s_co  = cooccurrence(a, b, via=story_links(version))   # 2) repeatedly linked via stories / a 3rd table
        s_emb = embedding_affinity(a, b)                # 3) consistently close in the shared space (F6)
        score = 0.45*s_val + 0.30*s_co + 0.25*s_emb
        if score >= CFG.relation_discovery:             # threshold in config, not code
            cands.append(RelationCandidate(a, b, round(score,3),
                          cardinality=infer_cardinality(a, b),
                          evidence=sample_links(a, b, n=3)))
    # Gate every candidate (identity, grounding, contradiction). A survivor becomes either a
    # relation_def proposal (admin-confirmed -> a real FK) or a KG Layer-B pending_edge - never auto-applied.
    return [c for c in cands if gate_pass(c, gates=["G1", "G5", "G6"])]

Done when. every relationship in the system is traceable to a relation_def, a link table, the crosswalk, the story bridge or the KG; and the discovery script surfaces a latent relationship (e.g. an offering scope that implicitly references subcaps) as a gated candidate rather than a silent edge.

def ingest_sow(file, version):
    safe = dlp_redact(file)                          # DLP BEFORE any model sees the document
    for row in normalise_scope_items(safe):          # one clean row per scope item
        item = enrich_scope_item(row)                # canonical scope-item record
        ev   = persist_evidence(item, kind="sow_chunk")   # carries account, opportunity, qty, assumptions
        for m in match_scope_to_subcaps(item, version):   # F6 shared embedding space
            if gate_pass(m, gates=["G1","G5","G7"]):
                persist_match(ev, m, status=("confirmed" if m.sim >= CFG.auto else "review"))
            # < CFG.review -> unmapped + nearest-neighbour proposal; queued, never dropped, never shown ungated

def enrich_scope_item(row):
    return ScopeItem(
        scope_name   = clean(row["Scope Item/Task Name"]),
        description  = clean(row["Scope Description"]),

The self-healing contract

16.4 Migration resilience across rebuilds and Terraform

  • State lives in the database. alembic_version - not the image, not Terraform state - is the source of truth, so rebuilding the image or re-running terraform apply changes nothing if the DB is already at head.
  • Idempotent by construction. upgrade head at head is a no-op; the version-check-first skip means most runs never even take the lock; Job retries are safe.
  • Serialized and bounded. One advisory lock on a direct connection with a 30s lock_timeout serializes any concurrent runners and prevents the pooler hang; a stuck holder cannot block deploys indefinitely.
  • Atomic. Postgres transactional DDL rolls a half-applied migration back, so a failed step never leaves a corrupt schema.
  • Zero-downtime shape. Expand/contract (add nullable -> backfill -> later drop) keeps the old and new revisions both working during the canary; CREATE INDEX CONCURRENTLY runs outside a transaction.
  • Terraform’s role. Terraform provisions infra and triggers the migration Job and the service deploy; it never embeds migrations, so an apply is safe to repeat.
  • Existing databases. Stamp/baseline an already-provisioned database to its current revision instead of re-creating, then migrate forward.

16.5 Cloud SQL connection handling

Connect via the Cloud SQL connector; keep a small per-instance pool with pool_pre_ping; because total connections are instances × pool_size, cap max-instances beneath the database connection limit and recycle the pool on SIGTERM. The migration runner uses a separate direct connection so the advisory lock holds.

One-line summary

Build amd64, gate traffic with a startup probe, handle SIGTERM in 10s, migrate as a one-shot Job (advisory lock on a direct connection, idempotent, transactional) before any traffic, canary then promote with auto-rollback, and let Terraform trigger - never embed - the migration. Rebuilds and re-applies become no-ops.

18.1 What counts as a trend (techniques, not keyword counts)

A trend is an acceleration of corroborated, novel signal, detected by combining several signals over rolling weekly windows - never raw keyword frequency:

  • Embedding clustering. Every enriched evidence item is embedded into the shared space and clustered (density-based, e.g. HDBSCAN), so topics are coherent meanings, not keyword buckets.
  • Burst / velocity. Per cluster, week-over-week acceleration (a z-score or Kleinberg-style burst), so a trend is a change in rate, not steady background.
  • Novelty / emergence. Distance of the cluster centroid from every mapped subcap embedding; far = emergent (a candidate net-new subcap), near = a descriptor revision or new use case.
  • Corroboration + source diversity. Tier-weighted count of independent sources, so five independent T1/T2 sources outrank one blog repeated five times; near-duplicates are collapsed first (ERS-linked).
  • Persistence / decay. Tracked across weeks with time decay, so a one-week spike fades and a sustained signal strengthens.
  • Statistical floors + gates. Minimum cluster size and source diversity, a significance threshold (config), then G2 (sufficiency), G3 (tier floor) and G6 (contradiction) - noise never surfaces.

# intelligence/detect_trends.py - a trend is an ACCELERATION of corroborated, NOVEL signal, not a keyword count.
# Runs weekly over the evidence corpus; every output is gated and feeds a self-learning loop.

def detect_trends(window_weeks=8):
    items = evidence_since(window_weeks)                 # news, vendor events, benchmarks (already enriched)
    items = dedupe(items, by="embedding", thresh=0.95)   # collapse syndicated near-duplicates first
    clusters = cluster_embeddings(items)                 # HDBSCAN in the shared space -> coherent topics (not keywords)
    out = []
    for c in clusters:
        if len(c) < CFG.min_cluster or sources_distinct(c) < CFG.min_sources:
            continue                                     # statistical floor: ignore thin / single-source noise
        velocity   = burst_score(c.weekly_counts)        # week-over-week acceleration (z-score / Kleinberg)
        diversity  = tier_weighted_source_diversity(c)   # 5 independent T1/T2 > one blog repeated 5x (ERS-linked)
        novelty    = 1 - max_cosine(c.centroid, subcap_embeddings())   # far from everything mapped = emergent
        persistence= weeks_present(c) / window_weeks      # sustained, not a one-week spike (time-decayed)
        score = 0.35*velocity + 0.30*diversity + 0.20*novelty + 0.15*persistence
        if score < CFG.trend_threshold:                  # threshold in CONFIG, recalibrated from feedback
            continue
        subcaps = retrieve_subcaps(c.centroid, k=5)       # express the trend in CATALOGUE terms (shared space)
        impact  = "net_new_subcap" if novelty > CFG.emergent_cut else "descriptor_revision"
        t = persist_trend(c, score, subcaps, impact, emergent=novelty > CFG.emergent_cut)
        if not gate_pass(t, gates=["G2", "G3", "G6"]):    # evidence sufficiency, tier floor, contradiction
            t.status = "review"
        out.append(t)
    return out

18.2 News intelligence and de-duplication

The weekly news job fetches via connectors and grounded search, collapses near-duplicate articles by embedding similarity (so syndication does not inflate a signal), classifies each item (Flash) into a magnitude and a catalogue-impact class, scores ERS from tier, recency, specificity and corroboration, and maps it to subcaps by retrieval - all before anything is surfaced, and only ever as a gated suggestion.

18.3 How the intelligence improves itself (self-learning)

Analyst decisions are the training signal. Promoting or dismissing a trend, confirming or rejecting a subcap mapping, and accepting or rejecting the resulting suggestion are all logged as labels that drive three loops:

  • Threshold recalibration. The burst, novelty and min-source thresholds (config, not code) are re-tuned against tracked precision and recall, so the detector adapts without a code change.
  • Contrastive fine-tuning. Confirmed and rejected mappings become positive/negative pairs that periodically fine-tune the embeddings and subcap retrieval (F6), so clustering and linkage get sharper.
  • Trend-quality ranker. A lightweight model learns which clusters tend to be acted on and orders them, so analysts see high-value trends first.

A weekly offline eval on a labelled set tracks trend precision (did surfaced trends get acted on?) and emergent recall (did we catch the net-new ones?); a drop triggers a retune. This is the same learning-loop discipline as F4 automap and F13.

# Structure forces specificity; guards reject copying or no-op changes; an adversary must be survived; gates decide.

def consultant_loop(seed):                               # seed = a trend, a news item, or analyst input
    # 1) Frame a thesis tied to a SPECIFIC subcap + change type (no vague "this matters").
    thesis = frame_thesis(seed, schema=ThesisSchema)     # {subcap_id, change_type, rationale, expected_maturity_lift}

    # 2) Retrieve grounding from the STORE (RAG). The model may reason ONLY over this (G5).
    ctx = retrieve_for_reasoning(thesis.subcap_id, thesis.query)   # subcap text + evidence + peer subcaps + stories

    # 3) Synthesise a DELTA (what should change vs the current text), citing evidence ids - not a summary.
    proposal = synthesize_delta(thesis, ctx, schema=DeltaSchema)   # {before, after, claims:[{text, evidence_id}], label}

    # ---- anti-generic safeguards: ALL must pass ----
    if extractive_overlap(proposal.after, ctx.sources) > CFG.copy_max:        # near-copy of a source?
        return reject("extractive: reproduces source text rather than synthesising")
    if similarity(proposal.after, ctx.subcap.descriptor) > CFG.redundant_max: # no real change?
        return reject("redundant: not a meaningful change to the existing descriptor")
    if any(claim.evidence_id is None for claim in proposal.claims):           # an ungrounded claim? (G5)
        return reject("ungrounded claim present")
    if contradicts_delivery(proposal, ctx.stories):                          # contradicts what we deliver? (G6)
        return reject("contradicts delivery evidence")

    # 4) Adversarial challenge - a second model argues the OPPOSITE; the thesis must survive.
    critique = adversary_review(proposal, ctx)           # Opus: missing evidence, overreach, weak sourcing
    if not critique.survives:
        return to_review(proposal, critique)

    # 5) Gate G1-G8 (esp. G2 sufficiency, G3 tier floor, G7 citations) and present with its reasoning chain.
    if gate_pass(proposal, gates=ALL_GATES):
        return as_suggestion(proposal, critique, chain=reasoning(thesis, ctx, proposal, critique))
    return to_review(proposal, critique)

18.5 How the AI accesses stored information and uses search

Reasoning reads the store; search only feeds the store. The AI retrieves from Postgres with hybrid retrieval (lexical tsvector + dense pgvector + structured SQL filters) over the active version, pulling the subcap record, the supporting evidence chunks with their source metadata, peer subcaps and delivery stories. Freshness comes from Gemini grounded Google Search and File Search, but those results are ingested as evidence, scored and gated before they can influence the catalogue - so the AI never reasons straight from the open web, and every claim cites a stored, retrievable evidence id (G5, G7).

# intelligence/retrieve.py - how the AI accesses stored knowledge and blends it with fresh search.
# Reasoning reads the STORE; grounded search only FEEDS the store (and is gated before it can influence anything).

def retrieve_for_reasoning(subcap_id, query, version=active_version()):
    # Hybrid retrieval in one engine over the ACTIVE version (F6): lexical + dense + structured filters.
    lexical = fulltext_search(query, version)                 # tsvector + GIN
    dense   = vector_search(embed(query), version, k=20)      # pgvector + HNSW (shared space)
    cands   = blend(lexical, dense, filters={"pillar": ..., "min_tier": CFG.tier_floor})
    return Context(
        subcap   = load_subcap(subcap_id, version),           # current descriptors, use cases, platforms
        evidence = [e for e in cands if e.kind in EVIDENCE],   # each with source {name,type,url,tier,ers,fetched_at}
        peers    = nearest_subcaps(subcap_id, k=5, version=version),
        stories  = delivery_stories(subcap_id, version),       # does delivery support the claim?
        sources  = [e.source for e in cands],
        citations= [e.id for e in cands])                      # every reasoning claim must cite one of these

18.6 Fine-tuning over time and cost

  • What is tuned, in order of leverage. (1) the shared embedding space (contrastive fine-tune on confirmed subcap-evidence and subcap-story pairs) - highest leverage; (2) optionally a supervised-tuned Flash model for classification/mapping on confirmed labels; (3) few-shot exemplars from confirmed mappings injected via cached context - the cheapest, immediate gain; (4) thresholds recalibrated from feedback (config, no training).
  • When. The weekly eval drives it: an MRR or precision drop triggers a retune; otherwise nothing is touched.
  • Cost. Prefer few-shot + caching (no training cost) until the eval justifies a tune; tune the smallest model that clears the bar; run tuning and all ingestion inference in Batch (50% off); cache the taxonomy and exemplars (~90% off); the weekly cadence keeps volume low; G8 and the cost meter keep spend inside the envelope.

Section 19

Global acceptance

These are the cross-cutting invariants every surface must satisfy, on top of its own per-page "Done when". They are the release checklist; each is enforced by a foundation, so meeting them is a property of the build, not a per-page effort.

  • Version-correct. Every catalogue read is scoped to the active version, every count/diff/lineage names its version, and switching the version refetches cleanly.
  • Intelligently mapped. A workbook with renamed columns still auto-maps from value fingerprints and learned memory; an admin override measurably improves the next automap (F4/F13).
  • One consistent corpus. Every story - seed or newly ingested from Jira - and every SOW scope item is enriched into the canonical shape before use; a story added today carries the same scored fields as a seed story.
  • Trust-visible. Every surface that shows an AI-derived value also shows the claim label, source tier, ERS and a working reasoning backlink.
  • Gated mutation. No catalogue edit, KG edge, SOW link or offering bundle is committed without passing G1-G8; an apply re-gates server-side and writes a versioned snapshot + an audit row.
  • Server-bounded. Heavy lists (14,406 stories, use cases, SOW scope items) paginate and filter on the server; read surfaces hold p95 < 1s and grounded chat p95 < 3s.
  • Provenance-complete. Nothing is silently dropped: unmatched stories, ungated SOW matches and gate failures are queued and visible, never discarded.
  • Admin-bounded. Upload, schema mapping, ingestion, knowledge graph, version revert, the source registry and the cost meter are gated by is_admin; login is the only pre-auth route.
  • Grounded. Chat and matches come only from retrieved catalogue + evidence (G5), never model memory; the fixed catalogue-grounded eval set passes before release.
  • Verifiable. Audit-grade exports are HMAC-signed and verify; the audit log is append-only.
  • Linkage-tested. Every page carries explicit frontend-to-backend QA checks - contract shape, version-keying, empty/large/error states, role gating and the trust envelope - and they pass.
  • Schema-explicit. Every relationship traces to a relation_def, a link table, the crosswalk, the story bridge or the KG; a discovered relationship is a gated proposal, never a silent edge (F15).
  • Model-resilient and bounded. Every Gemini call goes through the pinned, retrying gemini.py wrapper (429/5xx backoff, MAX_TOKENS raises the cap, safety blocks route to review); news and trends run weekly; and spend stays inside G8 plus the cost meter (section 17).
  • Trends are earned, not counted. A surfaced trend has cleared clustering, burst, novelty, source-diversity and persistence thresholds and the gates; the thresholds recalibrate from analyst feedback (section 18).
  • Suggestions synthesise, not restate. Every suggestion is a grounded before/after delta that passes the extractive-copy, redundancy, grounding and contradiction guards and an adversarial review before a human sees it (section 18).
  • Reasoning reads the store. The AI answers only from hybrid-retrieved catalogue + evidence (G5) with citations; grounded search feeds the store and is gated, never read straight into a conclusion (section 18).
  • Sources are measured. Source diversity per subcap, per subvertical and per pillar is tracked and thin coverage is flagged, not assumed; syndication is de-duplicated and tiers are weighted (section 18).
